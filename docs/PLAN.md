# AI-CMS — ИИ-платформа управления сайтами (замена CMS)

## Context

Соединяем две проверенные на реальных проектах идеи:

- **Siberian Motorbears**: для большого класса небольших сайтов CMS не нужна — сайтом управляют,
  разговаривая с ИИ-агентом обычным языком; сайт = папка статических HTML/CSS/JS, правки делает ИИ,
  публикация — `git push`. Видение SaaS агентства: свой LLM, своя хостинг-инфраструктура,
  поддомены 3-го уровня «в один клик».
- **kv12chat**: типовую динамику (auth, права, CRUD, realtime, файлы, push) можно не писать —
  её закрывает BaaS (Supabase) + PWA, self-host в РФ, vendor-neutral.

**Полное видение продукта** (роадмап): мультитенантная платформа, где нетехнический клиент ведёт
свой сайт через ИИ-чат (текст→голос), с живым превью и публикацией; статика расширяется типовой
динамикой (блог+комментарии, отзывы, формы, интернет-магазин) через BaaS, а сайт ставится на телефон
как PWA.

### Решения, принятые в обсуждении

1. **Свободный код, а не структурная палитра.** ИИ редактирует произвольные HTML/CSS/JS + картинки
   (как SM), без фиксированного набора компонентов/тем — именно палитра отталкивает в обычных CMS.
   Версионирование/undo/publish — это **git на тенанта**, рантайм — **отдача статики**.
2. **MVP — только статика.** Никакого пер-сайтового бэкенда в MVP. Вся динамика/BaaS
   (комментарии/отзывы/формы/магазин), PWA+push, кастомные домены — отдельные поздние фазы.
3. **LLM — self-hosted, готовая open-модель** (без дообучения в MVP). Требования: вести текстовый
   диалог + верстать статику. Дообучение своей модели — оптимизация на потом.
4. **Голос отложен** (STT/TTS — поздняя фаза). MVP — только текст.
5. **Vendor-neutral**, основной таргет — self-host в РФ.

### Чем MVP осознанно НЕ является
Не мультитенантная БД с RLS, не комментарии/корзина/формы, не PWA, не своя дообученная модель,
не голос, не кастомные домены клиентов. Всё это — в роадмапе ниже, после валидации ядра.

## MVP — архитектура

MVP = **«Siberian Motorbears как SaaS»**: мультитенантный хостинг статики + поддомены в один клик +
редактирование через чат с self-hosted кодовой моделью.

```
                  *.platform.ru   (wildcard DNS A → VPS; wildcard TLS DNS-01)
                            │ 443
                      ┌─────▼─────┐
                      │   CADDY   │  ← единственный публичный вход
                      └──┬─────┬──┘
   host = acme.platform.ru│     │ host = app.platform.ru (студия+превью, за авторизацией)
              ┌───────────▼──┐ ┌▼───────────────────┐
              │ СТАТИКА       │ │  STUDIO (Next.js)   │
              │ /srv/sites/   │ │  чат + live-превью  │
              │  acme/ (pub)  │ │  publish / undo     │
              └───────────────┘ └──┬──────────────┬──┘
                                   │              │
                          ┌────────▼───┐   ┌──────▼──────────────┐
                          │ control-DB │   │ LLM-SVC (vLLM)      │
                          │ Postgres:  │   │ Qwen2.5-Coder,      │
                          │ tenants,   │   │ OpenAI-совм.,       │
                          │ owners,    │   │ tool-calling        │
                          │ sessions   │   └─────────────────────┘
                          └────────────┘
   Файлы тенанта = git-репозиторий: draft (рабочее дерево) ↔ published (отдаётся Caddy).
   Наружу только Caddy. control-DB и LLM — внутренняя docker-сеть.
```

**Стек**: Caddy (TLS/роутинг); статика тенантов отдаётся напрямую Caddy; Studio — Next.js 16 (App
Router, TS strict) — единственный «динамический» сервис (чат-оркестрация, файловые/git-операции,
авторизация владельца, отдача draft-превью); control-plane Postgres (минимальный — реестр тенантов,
владельцы, журнал сессий); vLLM с готовой open coding-моделью. Всё в Docker Compose, связь по ENV.

### Подсистема A. Хранение сайта и цикл редактирования (git-based)

- **Каждый тенант = git-репозиторий статических файлов** (`/srv/tenants/<id>.git` + рабочее дерево).
  Картинки/ассеты лежат там же, версионируются вместе с кодом. Объектное хранилище не нужно.
- **Состояния**: `draft` (рабочее дерево, его правит агент, показывается в превью) и `published`
  (то, что Caddy отдаёт на `acme.platform.ru`).
- **«Заливай»**: коммит draft → деплой в `/srv/sites/<sub>/` (rsync/checkout, либо переключение
  симлинка на commit) — атомарно. **«Отмени»/«верни как было»**: `git checkout`/`revert` рабочего
  дерева; история = git log.
- **Превью**: Studio отдаёт draft рабочего дерева по `app.platform.ru/preview/<id>` (или
  `preview.<sub>`), `no-store`, за авторизацией владельца.

### Подсистема B. Агентный цикл редактирования (сердце MVP)

Это «Claude Code/Cursor для одной папки статики тенанта», но на self-hosted модели. Studio гоняет
tool-loop: пользователь пишет в чат → модель читает/правит файлы → превью обновляется → пользователь
говорит «заливай»/«отмени».

- **Инструменты модели** (исполняются сервером, не клиентом):
  `fs_list(dir)`, `fs_read(path)`, `fs_write(path, content)`, `fs_edit(path, old, new)`,
  `asset_add(path, image)`, `preview_refresh()`, `publish()`, `revert(target)`.
- **Песочница**: все файловые операции жёстко ограничены корнем тенанта (защита от path-traversal);
  агент тенанта A физически не видит файлы тенанта B. Модель не имеет произвольного shell — только
  перечисленные tools (опционально безопасный форматтер/линтер).
- **Контекст сессии** хранится (для продолжения диалога и, позже, для датасета дообучения).
- **Self-hosted модель**: готовая open coding-модель с tool-calling, ориентир **Qwen2.5-Coder**
  (32B при наличии 1×48GB / 14B как минимум на 1×24GB, AWQ-квант), сервинг **vLLM** (continuous
  batching, OpenAI-совм. tool-calling). Ollama — для локальной разработки без GPU. Абстракция
  провайдера (`LLMProvider`) сохраняется, чтобы подставлять сильную хостед-модель для сравнения/
  fallback и чтобы позже безболезненно перейти на свою дообученную.

### Подсистема C. Мультитенантность, поддомены, инфраструктура

- **Wildcard**: единожды DNS `A *.platform.ru → VPS`; Caddy с **DNS-01** challenge → один
  wildcard-сертификат `*.platform.ru` на всех. Новый тенант не трогает DNS/Caddy/сертификат.
- **Роутинг**: Caddy по host → либо статика `/srv/sites/<sub>/` (опубликованные сайты, быстро/дёшево),
  либо Studio (`app.`/`preview.`). Резолв `sub → tenant_id` — через control-DB (LRU-кэш в Studio).
- **«Один клик»**: запись в `tenants(sub, owner_id, status)` + инициализация git-репозитория со
  стартовым шаблоном; reserved-list (`www/api/app/studio/admin`); доступен сразу (wildcard).
- **Авторизация владельца**: лёгкая (один Postgres + сессии; можно переиспользовать GoTrue из опыта
  kv12chat, но полноценный Supabase в MVP не нужен — только control-plane состояние).
- **Деплой** (как kv12chat): `ssh … './deploy.sh'` (git pull → миграции control-DB →
  `docker compose build studio` → `up -d`); опциональный CI поверх. LLM-узел обновляется отдельно.
- **Безопасность свободного кода** (на твоём домене/IP): каждый тенант — отдельный origin-поддомен
  (изоляция SOP бесплатна); cookies Studio строго на `app.platform.ru`, не на `.platform.ru`;
  дефолтный CSP на статике тенантов; политика abuse-модерации (фишинг/малварь/майнеры на нашем домене
  → реальный риск блокировок/репутации). Это не «валидация HTML», а **изоляция + abuse** — реальная,
  но ограниченная работа; класс рисков «утечка данных между тенантами» в MVP отсутствует (общей БД,
  доступной сайтам, нет).

## Структура репозитория (предлагаемая)

```
ai-cms/
  apps/studio/                 Next.js 16: чат-редактор, превью, publish/undo, авторизация
    app/(studio)/...           UI владельца: чат + iframe-превью draft
    app/preview/[id]/...        отдача draft рабочего дерева (auth)
    lib/agent/loop.ts           tool-loop: сообщение → правки → превью
    lib/agent/tools.ts          fs_list/read/write/edit, asset_add, publish, revert (sandbox)
    lib/git/tenant-repo.ts      git-операции на репозитории тенанта (draft/publish/undo)
    lib/llm/provider.ts         абстракция vLLM/Ollama/хостед, OpenAI-совм. tool-calling
    lib/tenant.ts               резолв sub→tenant, провижининг «в один клик»
  control/migrations/          Postgres: tenants, owners, sessions (минимум)
  infra/                       docker-compose(.selfhost/.cloud).yml, Caddyfile, deploy.sh
  templates/                   стартовые шаблоны статического сайта (скелет для нового тенанта)
  eval/                        спайк-харнес: проверка модели на наборе SM-style правок
```

## MVP — последовательность реализации

0. **Спайк модели (главный риск, делаем первым).** Поднять vLLM с Qwen2.5-Coder, прогнать набор
   реальных SM-style запросов («сделай фон красным», «добавь пункт меню Контакты», «почини вёрстку
   длинного блока») через простой tool-loop на тестовой папке. Цель — убедиться, что готовая модель
   надёжно правит статику с tool-calling. Если 32B/14B слабоваты — пересмотреть выбор модели/GPU или
   рассмотреть гибрид с сильной моделью. **Гейт перед остальной работой.**
1. **Агентный цикл + git** на одной локальной папке: tool-loop, sandbox-tools, draft/publish/undo
   через git. Без мультитенантности и UI — CLI/скриптом.
2. **Studio-UI**: авторизация владельца, чат, iframe live-превью draft, кнопки publish/undo.
3. **Мультитенантность**: control-DB (tenants/owners/sessions), резолв sub→tenant, провижининг
   «в один клик», стартовый шаблон.
4. **Инфра self-host**: Caddy wildcard DNS-01 + роутинг (статика vs Studio), docker-compose,
   `deploy.sh`, бэкап git-репозиториев + control-DB.
5. **Хардненинг свободного кода**: CSP по умолчанию, изоляция origin/cookies, базовая abuse-политика.

## Роадмап (после MVP)

- **Динамика/BaaS**: Supabase (один инстанс, RLS по `tenant_id`), блог+комментарии, отзывы, формы,
  затем интернет-магазин. Стык с общим бэкендом — через **платформенные веб-компоненты/SDK**
  (`<aicms-comments post="…">`), которые безопасно делают tenant-scoping/RLS; вёрстка вокруг остаётся
  свободной (вопрос отложен пользователем — решить при входе в фазу).
- **PWA + Web Push** per-tenant (динамический manifest, service worker; push для e-commerce-статусов).
- **Голос**: STT (self-hosted Whisper) → тот же текстовый канал; опц. TTS-подтверждения.
- **Своя дообученная модель**: сбор датасета из реальных сессий (вход: файлы+запрос, выход:
  применённые правки; метки confirmed/reverted; PII обезличивать), QLoRA на open coding-модели,
  eval-гейт ≥ baseline, переключение в vLLM, готовая/хостед — в fallback.
- **Кастомные домены** клиента (Caddy on-demand TLS + `ask`-эндпоинт против verified-домена).
- **Гибрид-роутинг моделей** (small — простые правки, сильная — генерация нового).

## Ключевые риски (MVP)

- **Качество готовой модели на свободном коде** — главный риск → спайк (фаза 0) как гейт.
- **Песочница файловых операций** — агент не должен выйти за корень тенанта → строгая нормализация
  путей, тесты на traversal, изоляция процессов.
- **Abuse свободного кода на нашем домене** (фишинг/малварь) → изоляция origin + CSP + abuse-политика;
  при этом утечки данных между тенантами в MVP нет (нет общей доступной сайтам БД).
- **Блокировки в РФ** → self-host, не завязываться на `*.supabase.co`/`*.vercel.app` в проде.
- **GPU-узел недоступен** → деградация: показать ошибку/очередь; абстракция допускает временный
  fallback на хостед-модель.

## Верификация (MVP, end-to-end)

- **Спайк (фаза 0)**: на наборе из ~10 реальных SM-style запросов модель через tool-loop вносит
  корректные правки; меряем долю правок, дающих ожидаемый видимый результат без ручного вмешательства.
- **Агент+git**: unit/интеграционные — tools не выходят за корень тенанта (traversal-тесты);
  publish атомарен; undo/revert возвращает прежнее состояние (git).
- **E2E локально**: поднять стек (`docker compose` + vLLM/Ollama), создать тенанта «в один клик» →
  открыть Studio → в чате «сделай фон тёмным и добавь пункт меню Контакты» → увидеть в превью →
  publish → проверить опубликованную версию на `acme.localhost` → «верни как было» → откат.
- **Инфра**: новый поддомен доступен без правок DNS/Caddy (wildcard); статика отдаётся Caddy напрямую;
  Studio/preview — только за авторизацией.
