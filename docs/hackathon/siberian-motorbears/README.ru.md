# Статичный сайт под управлением ИИ — без CMS, без бэкенда, без сервера

> Кейс мастерской автодомов **Siberian Motorbears** (Новосибирск): рабочий
> коммерческий сайт, который проектирует, наполняет, переводит, оптимизирует под
> SEO и публикует один человек **без технических навыков** — задачи ставятся
> обычным текстом, всё остальное делает ИИ-агент в консоли (Claude Code / Cursor).

🇬🇧 English version: [README.en.md](README.en.md)

---

## TL;DR — главная мысль

Для огромного класса небольших сайтов (визитки, лендинги, сайты малого бизнеса,
блоги) **больше не нужны ни CMS, ни шаблонизатор, ни бэкенд, ни сервер**.
Достаточно:

1. **Статичные HTML/CSS/JS-файлы** — их хранит и отдаёт кто угодно.
2. **Бесплатный хостинг** — GitHub Pages + домен + DNS.
3. **ИИ-агент в консоли** как единственный «инструмент управления контентом».

Вёрстку, общие блоки (меню, подвал, хлебные крошки), сложные UI-компоненты,
интеграции, интернационализацию и SEO — всё это раньше брала на себя CMS с её
палитрой готовых компонентов. Теперь это описывается человеческим языком, а
реализует ИИ. Рынок «коробочных» CMS для простых сайтов, на мой взгляд, в
ближайшие годы радикально сожмётся.

## 1. Архитектура

```
Браузер ──► GitHub Pages (CDN) ──► статичные файлы из папки /site
                                     index.html, /routes, /blog, /en, ...
                                     css/, js/, assets/img/
```

- **Никакого сервера приложений и БД.** Весь сайт — это папка `site/`: ~300 HTML-файлов,
  один `css/main.css`, немного ванильного JS (карусели, карта, переключатель языка).
- **Хостинг:** GitHub Pages. При `push` в `main` GitHub Action копирует `site/` в
  корень Pages и публикует на `https://siberian-motorbears.ru`.
- **Домен и DNS** настраиваются один раз; в репозитории лежит `site/CNAME`.
- **«Шаблонизатора» нет** — единообразие общих блоков (шапка, меню, подвал)
  поддерживает агент: при правке он проходит по всем страницам и синхронизирует их.
  Где повторяющейся логики много (блог, английская версия, маршруты) — её
  генерируют небольшие Python-скрипты, которые **тоже написаны агентом**.

### Почему это работает
- Статику невозможно «положить» нагрузкой и почти нечего взламывать.
- Скорость и стоимость хостинга — около нуля.
- Весь сайт версионируется в git: любой контент-правкой можно откатиться.
- При этом на страницах есть вполне сложный UI: интерактивная карта маршрутов
  (Leaflet + геометрия треков), карусели фотографий, полноэкранный просмотр,
  адаптивная мобильная вёрстка, разметка JSON-LD, hreflang, sitemap.

## 2. Структура репозитория

| Путь | Назначение |
|------|------------|
| `site/` | **Сам сайт** — статичные файлы, которые публикуются на GitHub Pages. |
| `site/blog/` | Блог: данные записей (`entries/*.json` + медиа), `index.html`, индивидуальные страницы записей, манифест, заголовки, sitemap. |
| `site/routes/` | Раздел «Маршруты»: интерактивные карты Leaflet, геометрия треков. |
| `site/en/` | Английская копия всех основных страниц (тот же путь + префикс `/en/`). |
| `scripts/` | Python-утилиты, которыми оперирует агент: VK-синхронизация блога, генерация страниц, машинный/ручной перевод, деплой. |
| `agent_context/` | **«Память» агента между сессиями**: обзор проекта, журнал решений, договорённости. |
| `.cursor/rules/` | Правила-инструкции для ИИ (как обновлять блог, как деплоить, как добавлять маршруты). |
| `docs/` | Документация для человека: блог, SEO, сохранение Метрики/вебмастеров. |
| `.github/workflows/` | GitHub Action деплоя на Pages. |

## 3. Подход к разработке: «ИИ как разработчик»

Ключевая идея — **человек ставит задачу на родном языке, агент реализует её в коде**.
Чтобы это было устойчиво между сессиями, в репозитории есть три механизма «памяти»:

1. **`agent_context/`** — агент читает эту папку в начале работы: что за проект,
   какие приняты решения, какие соглашения по структуре и именованию.
2. **`.cursor/rules/*.mdc`** — короткие правила, привязанные к файлам (`globs`),
   которые подгружаются автоматически. Например, правило «при обновлении блога из
   VK сгенерируй осмысленные заголовки для двух свежих записей на двух языках».
3. **`docs/`** — человекочитаемая документация, на которую ссылаются и человек, и агент.

Так агент не «забывает» договорённости и не приходится каждый раз заново
объяснять контекст. Это и есть замена тому, что в CMS зашито в её модель данных и
админку, — только в виде обычного текста.

### Что именно делает агент
- Создаёт и правит страницы по текстовому описанию.
- Синхронизирует общие блоки на всех страницах.
- Пишет вспомогательные Python-скрипты под повторяющиеся задачи.
- Наполняет контент: ищет источники в интернете, пишет статьи в блог, считает
  инженерные оценки, подбирает фотографии и подписи.
- Поддерживает английскую версию и переводы.
- Следит за SEO: sitemap, canonical, hreflang, JSON-LD, метатеги.
- Деплоит сайт.

## 4. Управление контентом (рабочие процессы)

### 4.1. Блог, который сам подтягивается из соцсети (VK)
Блог компании ведётся в VK. Скрипты переносят его на сайт **по запросу вебмастера**:

```bash
python3 scripts/update_blog.py          # взять последние 5 постов из VK: добавить/обновить/удалить
python3 scripts/render_blog_entry_pages.py 200   # сгенерировать индивидуальные страницы + sitemap
python3 scripts/render_blog_initial.py  # перегенерировать вшитые в /blog/ посты
```

`update_blog.py` сравнивает посты VK с локальными `entry_*.json`, скачивает фото,
обновляет списки на странице блога и удаляет записи, которых больше нет в VK
(вместе с их HTML-страницей, манифестом и строкой в sitemap). Это «условно
динамический» контент, реализованный полностью статикой: данные превращаются в
готовые HTML один раз, при обновлении.

### 4.2. Интернационализация и перевод
Английская версия `/en/` собирается скриптами, а тексты переводятся:
- **массово и дёшево** — машинный перевод (`scripts/mt_html_lib.py`, MyMemory + кеш);
- **качественно** — вручную самим ИИ, без внешних API (реальный промпт вебмастера:
  *«На английский переводи сам "вручную", без внешних API»*). Даже динамические части
  (заголовки и тексты записей блога) переводятся и сохраняются в `text_en`/`entry_titles_en.json`.

### 4.3. Сложный UI без бэкенда + как ИИ сам проектирует маршруты
Раздел `site/routes/` — пример нетривиального компонента: интерактивные карты
Leaflet, маршруты, проложенные по реальным дорогам. И, что важнее для темы хакатона,
**это пример, где ИИ не просто верстает, а проектирует контент** — придумывает сам
маршрут по короткому запросу вебмастера.

Вебмастер пишет, например: *«добавь новый маршрут: Телецкое озеро через Чуйский
тракт, кольцом, ~3 дня»*. Дальше агент действует по правилу `.cursor/rules/routes.mdc`
и инструкции `site/routes/README-ROUTES.md`:

1. **Продумывает сам маршрут** — подбирает **реальные точки** (горнолыжные комплексы,
   заказники, конкретные достопримечательности и стоянки, а не «область вообще») и
   выстраивает их в порядке проезда как проезжаемое кольцо со стартом/финишем в
   Новосибирске.
2. **`build-route-geometries.js`** — добавляет трек `TRACKS.<key>` (последовательность
   координат `[широта, долгота]`).
3. **`node build-route-geometries.js <key>`** — запрашивает у **OSRM** геометрию
   маршрута **по дорогам** и записывает полилинии в `route-geometries.js`.
4. **`routes-maps.js`** — добавляет `ROUTES[<key>]`: центр, зум, трек и `waypoints`
   (маркеры с подписями и типом `sight`/`stop`).
5. **`index.html`** — карточку маршрута, пункт легенды, `ROUTE_INFO`, расстояние и
   план по дням; ключ добавляется в `ROUTE_ORDER` **с учётом сортировки по
   длительности** (короткие выше длинных), причём тематический «Горнолыжное кольцо
   Сибири» (`ski_ring`) всегда остаётся последним.

Один и тот же `key` агент аккуратно проводит через все файлы (`data-route`, `TRACKS`,
`ROUTES`, `ROUTE_GEOMETRIES`, `ROUTE_INFO`, `ROUTE_ORDER`), а интерактивная карта на
странице собирается сама из этих данных. Никакого бэкенда — только статичные файлы и
один разовый запрос к OSRM на этапе сборки геометрии.

### 4.4. SEO и аналитика
В `docs/` зафиксировано, как при обновлении сохранять Яндекс.Метрику, Яндекс.Вебмастер,
Google Search Console и VK Пиксель (одни и те же счётчики и проверочные метки/файлы).
Агент при добавлении страниц обновляет `sitemap.xml`, canonical, hreflang и JSON-LD.

## 5. Деплой

```bash
./scripts/deploy-github.sh "Краткое описание правок"
```

Скрипт берёт учётные данные из `.env`/`scripts/.env` (`GITHUB_USER`, `GITHUB_TOKEN`),
делает `git add -A`, коммит и `git push origin main`. После пуша срабатывает
`.github/workflows/deploy-pages.yml`: копирует `site/` в корень Pages и публикует.
Через 1–2 минуты сайт обновлён. Сервер настраивать не нужно вообще.

## 6. Примеры промптов вебмастера

Это **реальные** запросы из истории работы над проектом — ими и управляется сайт:

**Контент блога**
- «Изучи источники на разных языках и составь новую запись для блога по теме
  "классификация автодомов". Опиши разные подходы к классификации…»
- «Статья нужна в plain text. Без форматирования, без таблиц. Картинки сохрани
  отдельно, и на каждой подпиши — какой это класс.»
- «Ты в статье слишком много врёшь о том, что мы "уже умеем". На самом деле наша
  первая зимняя модель — это R&D. Перепиши честно.»
- «Давай посчитаем и добавим в статью: сколько килограммов добавляет каждый
  сантиметр утеплителя ЭППС для фургона заданного размера.»

**Обновление из VK + перевод**
- «обнови блог — в VK появилась новая запись. На английский переводи сам "вручную",
  без внешних API.»

**Маршруты (ИИ проектирует контент)**
- «добавь новый маршрут: Телецкое озеро через Чуйский тракт, кольцом, ~3 дня» —
  агент сам подбирает точки, строит геометрию по дорогам через OSRM и собирает карточку.
- «найди фотку получше для fifth-wheel.»

**Вёрстка и UI**
- «Последняя длинная запись в блоге ломает вёрстку. Сделай так, чтобы блок текста
  по высоте не превышал блок с фото и видео справа.»
- «Уменьши размер шрифта в секции "Все записи". Если оставить только заголовки,
  можно ужать ещё и разложить в 2–3 колонки.»
- «Mobile header: меньше отступы, сворачивай закрытое меню.»

**SEO**
- «Google Search Console сообщает, что много страниц не проиндексировано. Разберись
  почему» (далее агент анализировал выгрузки GSC и чинил sitemap/canonical/hreflang).

**Деплой**
- «deploy to github» / «подними сайт локально».

## 7. Предложение: автоматизация синхронизации с VK по расписанию

Сейчас блог обновляется **вручную по команде**. Это можно автоматизировать
облачным агентом по расписанию. Варианты (от простого к «умному»):

1. **GitHub Actions по cron (механическая часть).**
   Запланированный workflow раз в сутки запускает `scripts/update_blog.py` и, если
   есть изменения, коммитит и пушит. VK-токен — в Secrets репозитория. Деплой
   срабатывает автоматически. Минус: машинная часть (скачивание) — без «умных»
   заголовков и качественного перевода.

2. **Запланированный ИИ-агент (умная часть).**
   Claude Code умеет запускать **scheduled remote agents** (routines по cron — см.
   навык `/schedule`). Такой агент по расписанию выполняет ровно тот же сценарий,
   что и вебмастер: `update_blog.py` → придумать заголовки для свежих записей →
   перевести их на английский «вручную» → `render_blog_entry_pages.py 200` →
   `render_blog_initial.py` → `deploy-github.sh`. То есть весь рабочий процесс из
   `.cursor/rules/blog-vk-update.mdc`, но без участия человека.

3. **Гибрид (рекомендуется).**
   Cron-workflow делает дешёвую механику ежедневно; «умный» агент запускается реже
   (или по факту появления нового поста — через webhook/RemoteTrigger) и
   дорабатывает заголовки и переводы. Так платный ИИ-вызов случается только когда
   реально есть новый контент.

> Триггер «появился новый пост в VK» можно повесить на тот же cron-агент (он сам
> проверяет ленту) или на внешний webhook, дёргающий remote-trigger агента.

## 8. Вывод для outstaff/outsource-агентств

- Для простых сайтов **CMS перестаёт быть нужной**: её роль (вёрстка из блоков,
  единые шаблоны, контент-формы, i18n, SEO) закрывается связкой «статика + ИИ-агент».
- **Порог входа для клиента — почти ноль**: управление = текстовые запросы.
  Это открывает огромный рынок малых сайтов, который сегодня обслуживают CMS.
- Агентству это меняет экономику: вместо «сверстать в CMS и поддерживать» —
  «настроить репозиторий, правила для агента и хостинг», а дальше клиент или
  агент-по-расписанию ведёт сайт сам.
- Сложный UI и интеграции **не являются стоп-фактором**: они реализуются как
  статичные компоненты и скрипты, которые пишет тот же агент.
