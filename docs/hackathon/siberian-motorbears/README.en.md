# An AI-managed static website — no CMS, no backend, no server

> A case study from the motorhome workshop **Siberian Motorbears** (Novosibirsk,
> Russia): a real commercial website that is designed, filled with content,
> translated, SEO-optimized and published by a single person **with no technical
> skills** — tasks are written in plain language, and everything else is done by an
> AI agent in the console (Claude Code / Cursor).

🇷🇺 Русская версия: [README.ru.md](README.ru.md)

---

## TL;DR — the core idea

For a huge class of small sites (business cards, landing pages, small-business
sites, blogs) you **no longer need a CMS, a templating engine, a backend, or a
server**. All you need is:

1. **Static HTML/CSS/JS files** — anything can store and serve them.
2. **Free hosting** — GitHub Pages + a domain + DNS.
3. **An AI agent in the console** as the one and only "content management tool".

Markup, shared blocks (menu, footer, breadcrumbs), complex UI components,
integrations, internationalization and SEO — all the things a CMS used to provide
through its palette of pre-built components — are now described in human language
and implemented by the AI. The market for "boxed" CMS products aimed at simple
sites will, in my view, shrink dramatically over the next few years.

## 1. Architecture

```
Browser ──► GitHub Pages (CDN) ──► static files from /site
                                    index.html, /routes, /blog, /en, ...
                                    css/, js/, assets/img/
```

- **No application server, no database.** The whole site is the `site/` folder:
  ~300 HTML files, one `css/main.css`, and a little vanilla JS (carousels, map,
  language switcher).
- **Hosting:** GitHub Pages. On `push` to `main`, a GitHub Action copies `site/`
  into the Pages root and publishes it at `https://siberian-motorbears.ru`.
- **Domain & DNS** are configured once; `site/CNAME` lives in the repo.
- **There is no "templating engine"** — consistency of shared blocks (header, menu,
  footer) is maintained by the agent: when something changes, it walks every page
  and synchronizes them. Where repetitive logic is heavy (blog, English version,
  routes), it is generated by small Python scripts that **were also written by the agent**.

### Why it works
- Static content is almost impossible to overload and has almost nothing to hack.
- Hosting speed and cost are near zero.
- The whole site is versioned in git: any content edit can be rolled back.
- And yet the pages carry genuinely complex UI: an interactive routes map
  (Leaflet + track geometry), photo carousels, fullscreen viewer, responsive
  mobile layout, JSON-LD markup, hreflang, sitemap.

## 2. Repository layout

| Path | Purpose |
|------|---------|
| `site/` | **The site itself** — static files published to GitHub Pages. |
| `site/blog/` | Blog: entry data (`entries/*.json` + media), `index.html`, per-entry pages, manifest, titles, sitemap. |
| `site/routes/` | "Routes" section: interactive Leaflet maps, track geometry. |
| `site/en/` | English copy of all main pages (same path + `/en/` prefix). |
| `scripts/` | Python utilities the agent drives: VK blog sync, page generation, machine/manual translation, deploy. |
| `agent_context/` | **The agent's cross-session memory**: project overview, decision log, conventions. |
| `.cursor/rules/` | Instruction rules for the AI (how to update the blog, how to deploy, how to add routes). |
| `docs/` | Human docs: blog, SEO, preserving analytics/webmaster verification. |
| `.github/workflows/` | GitHub Action that deploys to Pages. |

## 3. Development approach: "the AI is the developer"

The key idea: **the human states the task in plain language, and the agent
implements it in code.** To keep this stable across sessions, the repo carries
three "memory" mechanisms:

1. **`agent_context/`** — the agent reads this folder at the start of work: what
   the project is, which decisions were made, what the structure and naming
   conventions are.
2. **`.cursor/rules/*.mdc`** — short rules attached to files (`globs`) that load
   automatically. For example: "when updating the blog from VK, generate meaningful
   titles for the two newest entries in both languages."
3. **`docs/`** — human-readable documentation referenced by both the human and the agent.

This way the agent doesn't "forget" agreements and you don't re-explain context
every time. It's the replacement for what a CMS hard-codes into its data model and
admin panel — only expressed as plain text.

### What the agent actually does
- Creates and edits pages from a text description.
- Synchronizes shared blocks across all pages.
- Writes helper Python scripts for repetitive tasks.
- Produces content: researches sources online, writes blog posts, runs engineering
  estimates, picks photos and captions.
- Maintains the English version and translations.
- Handles SEO: sitemap, canonical, hreflang, JSON-LD, meta tags.
- Deploys the site.

## 4. Content management (workflows)

### 4.1. A blog that syncs itself from social media (VK)
The company blog is kept on VK. Scripts pull it onto the site **on the webmaster's request**:

```bash
python3 scripts/update_blog.py                  # take last 5 VK posts: add/update/delete
python3 scripts/render_blog_entry_pages.py 200  # generate per-entry pages + sitemap
python3 scripts/render_blog_initial.py          # regenerate posts embedded in /blog/
```

`update_blog.py` compares VK posts with local `entry_*.json`, downloads photos,
updates the lists on the blog page, and removes entries that no longer exist on VK
(together with their HTML page, manifest record, and sitemap line). This is
"quasi-dynamic" content delivered as pure static: data is turned into ready HTML
once, at update time.

### 4.2. Internationalization and translation
The English `/en/` version is assembled by scripts, while the text is translated:
- **in bulk and cheaply** — machine translation (`scripts/mt_html_lib.py`, MyMemory + cache);
- **with quality** — by the AI itself by hand, with no external API (a real
  webmaster prompt: *"translate to English yourself, by hand, without any external
  API"*). Even the dynamic parts (blog titles and bodies) are translated and stored
  in `text_en` / `entry_titles_en.json`.

### 4.3. Complex UI with no backend + how AI designs routes itself
`site/routes/` is an example of a non-trivial component: interactive Leaflet maps and
routes drawn along real roads. And — more relevant to the hackathon theme — **it's a
case where the AI doesn't just build markup, it designs the content**: it invents the
route itself from a short webmaster request.

The webmaster writes, for example: *"add a new route: Lake Teletskoye via the Chuysky
tract, as a loop, ~3 days."* The agent then follows `.cursor/rules/routes.mdc` and the
guide in `site/routes/README-ROUTES.md`:

1. **Designs the route itself** — it picks **real waypoints** (ski resorts, nature
   reserves, specific landmarks and stops — not "the region in general") and sequences
   them in driving order as a drivable loop starting and ending in Novosibirsk.
2. **`build-route-geometries.js`** — adds the track `TRACKS.<key>` (a sequence of
   `[lat, lng]` coordinates).
3. **`node build-route-geometries.js <key>`** — asks **OSRM** for the route geometry
   **along roads** and writes the polylines to `route-geometries.js`.
4. **`routes-maps.js`** — adds `ROUTES[<key>]`: center, zoom, track, and `waypoints`
   (annotated markers typed `sight`/`stop`).
5. **`index.html`** — the route card, a legend entry, `ROUTE_INFO`, distance and a
   day-by-day plan; the key goes into `ROUTE_ORDER` **respecting the sort by trip
   length** (shorter above longer), while the themed "Siberian ski loop" (`ski_ring`)
   always stays last.

The agent carefully threads the same `key` through every file (`data-route`, `TRACKS`,
`ROUTES`, `ROUTE_GEOMETRIES`, `ROUTE_INFO`, `ROUTE_ORDER`), and the interactive map on
the page assembles itself from that data. No backend — just static files and a single
one-off OSRM request when the geometry is built.

### 4.4. SEO and analytics
`docs/` records how to preserve Yandex.Metrica, Yandex.Webmaster, Google Search
Console and the VK pixel across updates (same counters and verification tags/files).
When adding pages, the agent updates `sitemap.xml`, canonical, hreflang and JSON-LD.

## 5. Deployment

```bash
./scripts/deploy-github.sh "Short description of changes"
```

The script reads credentials from `.env`/`scripts/.env` (`GITHUB_USER`,
`GITHUB_TOKEN`), runs `git add -A`, commits and `git push origin main`. The push
triggers `.github/workflows/deploy-pages.yml`, which copies `site/` to the Pages
root and publishes. The site is live in 1–2 minutes. No server to configure at all.

## 6. Webmaster prompt examples

These are **real** requests from the project history — this is literally how the
site is managed:

**Blog content**
- "Research sources in different languages and write a new blog post on 'motorhome
  classification'. Describe the different approaches to classifying campervans…"
- "I want the article as plain text. No formatting, no tables. Save the images
  separately and caption each one with its class."
- "You're overclaiming in the article about what we 'already do'. Our first winter
  model is actually R&D. Rewrite it honestly."
- "Let's also calculate and add to the article: how many kilograms each centimeter
  of XPS insulation adds for a van of a given size."

**VK sync + translation**
- "update the blog — there's a new post on VK. Translate to English yourself, by
  hand, without any external API."

**Routes (AI designs the content)**
- "add a new route: Lake Teletskoye via the Chuysky tract, as a loop, ~3 days" — the
  agent picks the waypoints itself, builds road geometry via OSRM, and assembles the card.
- "find a better photo for the fifth-wheel."

**Layout and UI**
- "The latest long blog entry breaks the layout. Make the text block no taller than
  the photo/video block on the right."
- "Reduce the font size in the 'All posts' section. If we keep only titles, we can
  shrink it further and use 2–3 columns."
- "Mobile header: less padding, collapse the closed nav."

**SEO**
- "Google Search Console reports many pages aren't indexed. Figure out why" (the
  agent then analyzed the GSC exports and fixed sitemap/canonical/hreflang).

**Deploy**
- "deploy to github" / "spin the site up locally".

## 7. Proposal: scheduling the VK sync automatically

Today the blog is updated **manually, on command**. This can be automated with a
scheduled cloud agent. Options, from simple to "smart":

1. **GitHub Actions cron (the mechanical part).**
   A scheduled workflow runs `scripts/update_blog.py` once a day and, if there are
   changes, commits and pushes. The VK token lives in repo Secrets. Deploy fires
   automatically. Downside: only the mechanical part (downloading) — no "smart"
   titles or quality translation.

2. **A scheduled AI agent (the smart part).**
   Claude Code can run **scheduled remote agents** (cron routines — see the
   `/schedule` skill). Such an agent runs exactly the same playbook as the
   webmaster: `update_blog.py` → invent titles for the new entries → translate them
   to English "by hand" → `render_blog_entry_pages.py 200` → `render_blog_initial.py`
   → `deploy-github.sh`. That is the whole workflow from
   `.cursor/rules/blog-vk-update.mdc`, but with no human in the loop.

3. **Hybrid (recommended).**
   The cron workflow does the cheap mechanics daily; the "smart" agent runs less
   often (or on the appearance of a new post — via webhook/RemoteTrigger) and
   refines titles and translations. That way the paid AI call happens only when
   there is actually new content.

> The "new VK post" trigger can be handled by the same cron agent (it polls the
> feed itself) or by an external webhook hitting a remote-trigger for the agent.

## 8. Takeaway for outstaff/outsource agencies

- For simple sites the **CMS stops being necessary**: its role (block-based markup,
  shared templates, content forms, i18n, SEO) is covered by "static + AI agent".
- **The barrier to entry for the client is almost zero**: management = text
  requests. This opens the huge small-site market currently served by CMS products.
- It changes an agency's economics: instead of "build it in a CMS and maintain it",
  it becomes "set up a repo, agent rules and hosting", after which the client — or a
  scheduled agent — runs the site themselves.
- Complex UI and integrations are **not a blocker**: they are implemented as static
  components and scripts written by the same agent.
