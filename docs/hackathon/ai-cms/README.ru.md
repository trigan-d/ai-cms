# AI-CMS — сайт под управлением разговора с ИИ

> Замена традиционной CMS: нетехнический клиент создаёт и ведёт свой небольшой сайт,
> просто **разговаривая** (текстом или голосом) с ИИ-агентом. Никаких админок, тем,
> плагинов и конструкторов из блоков — только разговор и результат в живом превью.

🇬🇧 English version: [README.en.md](README.en.md)

---

## TL;DR — главная мысль

Для огромного класса небольших сайтов (визитки, лендинги, сайты малого бизнеса, блоги)
не нужна CMS с палитрой готовых блоков. Достаточно дать клиенту **чат с ИИ**, который
правит сайт по человеческому описанию, показывает изменения в реальном времени и
публикует по команде «заливай». Платформа мультитенантная: каждый клиент получает сайт
на поддомене 3-го уровня (`мойсайт.platform.ru`) «в один клик», с автоматическим HTTPS.

Это синтез двух идей, проверенных на реальных проектах:
- **Siberian Motorbears** — сайт это папка статики, а «CMS» это ИИ-агент в консоли,
  который правит код по запросу на родном языке.
- **kv12chat** — типовой бэкенд можно не писать: его роль закрывает BaaS (Supabase), а
  нативное приложение заменяет PWA; всё self-host, vendor-neutral, работает в РФ.

---

## 1. Что отличает подход

В обычных CMS клиент собирает страницу из готовых блоков и упирается в их потолок.
Здесь **нет фиксированной палитры**: ИИ редактирует произвольный код (любой HTML/CSS/JS
+ картинки). «Сделай фон тёмным», «добавь страницу Контакты и пункт меню», «увеличь
заголовок», «свёрстай прайс в три колонки» — агент сам пишет нужную вёрстку под задачу,
а не подгоняет её под чужой плагин.

При этом сложное, что в CMS зашито в админку, тут закрыто инфраструктурой:
- **версии, отмена, «верни как было»** — это git на каждого тенанта;
- **публикация** — атомарное копирование в раздаваемую папку;
- **поддомены, HTTPS, мультитенантность** — Caddy + Supabase + middleware.

## 2. Архитектура

```
                *.platform.ru   (wildcard DNS + wildcard TLS)
                          │ 443
                    ┌─────▼─────┐  ← единственный публичный вход
                    │   CADDY   │  TLS, security-заголовки
                    └──┬─────┬──┘
   platform.ru / app   │     │ *.platform.ru (сайты тенантов) · api.platform.ru
              ┌─────────▼─┐ ┌─▼──── KONG ───────────────┐
              │  STUDIO   │ │ GoTrue · PostgREST · ...   │  (Supabase)
              │ Next.js16 │ └──────────────┬────────────┘
              │ чат+превью│                │
              │ proxy.ts  │         ┌──────▼──────┐
              └──┬─────┬──┘         │  POSTGRES   │ ← все тенанты, изоляция = RLS
                 │     │            │  + RLS      │
          ┌──────▼─┐ ┌─▼────────┐   └─────────────┘
          │ OLLAMA │ │ (kong)   │
          │  LLM   │ └──────────┘
          └────────┘
   Сайт тенанта = git-репозиторий: draft (правит агент) ↔ published (раздаётся).
```

**Стек:** Next.js 16 (App Router, RSC, TypeScript strict), Supabase self-hosted
(Postgres + GoTrue + PostgREST + Realtime + Storage + RLS), Ollama для self-hosted LLM,
Caddy (TLS/роутинг), всё в Docker Compose. Связь с сервисами — только через ENV,
поэтому тот же код запускается локально, в облаке и на своём VPS в РФ (vendor-neutral).

## 3. Как устроено редактирование (сердце проекта)

Это «Claude Code / Cursor для папки статики одного клиента», но на self-hosted модели:

1. Клиент пишет (или говорит) запрос в чат.
2. Агент в цикle вызывает инструменты: `fs_list`, `fs_read`, `fs_write`, `fs_edit`,
   `publish`, `revert`. Файловые операции **заперты в корне тенанта** (песочница,
   защита от выхода за пределы) — агент одного клиента физически не видит чужие файлы.
3. Чтобы агент не угадывал структуру, ему в системный промпт кладётся **текущее
   состояние сайта** (`buildSiteContext`) — это резко повышает точность правок.
4. Изменения попадают в **draft** и сразу видны в живом превью.
5. «Заливай» → атомарная публикация (commit + копирование в раздаваемую папку).
   «Отмени» / «верни как было» → откат через git.

**Как это реализовано (git под капотом).** Каждый сайт — это настоящий git-репозиторий, у
тенанта две папки: `data/tenants/<id>/` — рабочее дерево, которое правит агент (**черновик**,
он же в превью), и `data/sites/<поддомен>/` — опубликованная копия (её отдаёт сервер).
Сопоставление с git: создание сайта = `git init` + первый коммит; «Опубликовать» = `git commit`
(**новая версия**) + атомарное копирование в раздаваемую папку; «Отменить черновик» =
`git checkout`/`clean`; «верни как было» = forward-`git revert` (история сохраняется); список
версий = `git log`. Своей «системы версий» писать не пришлось — её роль закрывает git, а контент
живёт как обычные файлы. В Postgres только control-plane (владелец, поддомен, статус), не
содержимое сайтов.

**LLM — self-hosted.** Используется готовая open-модель с tool-calling; по умолчанию
`llama3.1:8b` (надёжно доводит многошаговые правки — создать страницу *и* добавить
ссылку в меню). Провайдер абстрагирован (OpenAI-совместимый эндпоинт: Ollama/vLLM или
хостед), поэтому модель можно заменить или дообучить под задачу.

## 4. Мультитенантность и динамика без бэкенд-кода

- **Один Supabase-инстанс, изоляция через `tenant_id` + RLS** (не «проект на клиента»).
  Владелец видит и правит только свои сайты — это гарантирует БД, а не код контроллера.
- **Поддомен «в один клик»**: запись в `tenants` + инициализация git-репозитория из
  шаблона. Wildcard-DNS и wildcard-TLS уже покрывают всех — DNS/Caddy не трогаются.
- **Аутентификация** — GoTrue (email+пароль), cookie-сессии через `@supabase/ssr`
  (паттерн из kv12chat). Самостоятельная регистрация клиента.

## 5. Фичи (реализовано)

- **ИИ-редактор**: чат + живое превью draft + «Опубликовать» / «Отменить черновик».
- **Свободный код**: любой HTML/CSS/JS, многостраничные сайты с навигацией.
- **Мультитенантность**: дашборд сайтов владельца, провижининг по поддомену, RLS-изоляция.
- **Раздача по поддомену**: `имя.platform.ru` отдаёт опубликованный сайт с авто-HTTPS.
- **Голос (R3)**: кнопка-микрофон — локально через Web Speech API браузера, в проде —
  self-hosted Whisper (`/api/stt`, OpenAI-совместимый).
- **Кастомные домены (R5)**: клиент привязывает свой домен (`www.магазин.рф`); Caddy
  выпускает сертификат on-demand только для проверенных доменов (`/api/tls/allowed`).
- **Хардненинг**: security-заголовки/CSP на отдаче сайтов, host-only cookie (сессия
  владельца не утекает на сайты тенантов), песочница файловых операций.
- **Инфраструктура**: Docker Compose (Supabase + Studio + Ollama + Caddy), деплой без CI
  (`deploy.sh`: git pull → миграции → пересборка → up), vendor-neutral.

## 6. Дальнейшие шаги

- **Динамика / BaaS (R1):** блог с комментариями, отзывы, формы — таблицы с `tenant_id`
  и RLS (паттерн уже отработан); запись от анонимов через тонкий серверный прокси с
  анти-спамом; стык с вёрсткой — через платформенные веб-компоненты (`<aicms-comments>`),
  чтобы свобода вёрстки сохранялась. Затем — интернет-магазин (товары, корзина, заказы).
- **PWA на тенанта (R2):** авто-manifest/service-worker/иконка и установка сайта клиента
  на телефон посетителя (код-заготовка есть).
- **Своя дообученная модель (R4):** датасет из реальных сессий редактирования
  (вход: файлы+запрос, выход: применённые правки) → дообучение небольшой модели (QLoRA)
  → переключение в инференс, готовая/хостед в fallback. Цель — себестоимость и работа в РФ.
- **Прод-деплой**: VPS в РФ, wildcard-DNS, Caddy с DNS-плагином провайдера; для скорости
  правок — GPU под инференс.
- **Бизнес-модель**: подписка «сайт, которым управляешь голосом»; у агентства почти
  нулевые предельные издержки — свой LLM, своя статика, свой деплой.

## Как запустить локально

```bash
# 1. Модель: Ollama + модель
ollama serve && ollama pull llama3.1:8b
# 2. Control-plane: Supabase (порты 553xx)
supabase start
# 3. Studio
pnpm --filter @ai-cms/studio dev        # → http://localhost:3000
# Опубликованные сайты — на http://<поддомен>.localhost:3000
```
Подробности и статус — в [`../STATUS.md`](../STATUS.md) и [`../../CLAUDE.md`](../../CLAUDE.md).
