# kv12chat — архитектура, разработка и эксплуатация

> Презентационный документ для хакатона. Аудитория — инженеры outstaff/outsource-агентства.
> Главный тезис: **для типового продукта отдельный бэкенд можно не писать — его роль закрывает Supabase (BaaS), а нативное приложение заменяет PWA.** Ниже — как это устроено на реальном живом проекте.

English version: [`architecture.en.md`](./architecture.en.md)

---

## 1. Что это за проект

**kv12chat** — приватный чат для жителей коттеджного посёлка «Квартал-12». Похож на WhatsApp/Telegram, но на своём сервере. Масштаб: до ~1000 пользователей, ≤1000 сообщений в день. Прод поднят 26 мая 2026 на одном VPS Selectel (Новосибирск) под доменом `kv12chat.ru`.

Функционал: каналы, сообщения в реальном времени, треды-«обсуждения», реакции (лайки), опросы, вложения-картинки, @-упоминания, редактирование/удаление, поиск, карта посёлка с участками и жильцами, заявки на регистрацию + админка, web-push, установка как PWA на телефон.

Стек (зафиксирован):

- **Frontend / API**: Next.js 16 (App Router, React 19, Server Components по умолчанию), TypeScript strict, Tailwind 4.
- **Backend = Supabase (self-hosted)**: Postgres, GoTrue (Auth), PostgREST (REST поверх БД), Realtime (WebSocket), Storage, Edge Runtime (Deno-функции).
- **Push**: Web Push API + VAPID, рассыльщик — Edge Function, дёргается из `pg_cron` каждые 10 секунд.
- **Хостинг**: один VPS, весь стек в Docker Compose, TLS — Caddy + Let's Encrypt. Никаких managed-сервисов, CI и vendor-lock.

## 2. Главная идея: Supabase вместо самописного бэкенда

В типовом проекте бэкенд — это связка «авторизация + права доступа + CRUD-API + realtime + файлы + фоновые задачи + уведомления». Каждый кусок обычно пишется руками. Supabase даёт всё это готовыми кирпичами поверх **одной базы Postgres**:

| Задача «классического» бэкенда | Как её закрывает Supabase в kv12chat |
| --- | --- |
| Регистрация и аутентификация | **GoTrue**: хранит `auth.users`, выдаёт JWT. Вход по email **или** телефону — резолвим идентификатор в email на сервере и зовём `signInWithPassword`. Своего auth-сервиса нет. |
| Авторизация (кто что видит) | **RLS-политики Postgres**: `auth.uid()` + флаги `is_admin`/`is_banned` проверяются прямо в БД на каждый запрос. Невозможно «забыть проверку» в контроллере. |
| CRUD-API к данным | **PostgREST**: REST-эндпоинты генерируются из схемы. Где нужна логика — тонкие Route Handlers Next.js или Postgres RPC. |
| Реалтайм-доставка | **Realtime**: подписка на `postgres_changes` (INSERT/UPDATE/DELETE) по таблицам `messages`, `reactions`, `poll_votes` через WebSocket. |
| Файлы и картинки | **Storage**: бакеты `avatars` (public) и `files` (private + signed URL). Весь доступ — через обёртку `@/lib/storage.ts`. |
| Фоновые задачи / cron | **pg_cron**: каждые 10 с дёргает Edge Function-рассыльщик push. |
| Очередь уведомлений | Таблица `notification_queue` + триггеры fan-out на вставку сообщения/упоминания/ответа в треде. |
| Полнотекстовый поиск | Postgres FTS (`tsvector`, русский словарь) + GIN-индекс; для автодополнения имён — `pg_trgm`. |
| Миграции схемы | SQL-файлы в `supabase/migrations/`, накатываются на деплое. |

**Вывод**: в проекте практически нет «бэкенд-кода» в привычном смысле. Есть Next.js (UI + тонкие API-роуты) и Postgres (схема + RLS + триггеры). Supabase — это и есть бэкенд.

## 3. Топология

```
Жители ─► Caddy (TLS Let's Encrypt) ─┬─► nextjs:3000        (kv12chat.ru)
                                     └─► kong:8000          (api.kv12chat.ru)
                                            ├─ auth      (GoTrue)
                                            ├─ rest      (PostgREST)
                                            ├─ realtime  (WebSocket / WSS)
                                            ├─ storage
                                            └─ functions (push-dispatcher, Deno)
                                                   │
                                                   ▼
                                                 Postgres
                                                 + pg_cron каждые 10с → push-dispatcher
```

Всё — контейнеры в одном `infra/docker-compose.yml`. Наружу открыт только Caddy (80/443); все сервисы Supabase общаются по внутренней docker-сети.

Тонкость self-host, на которую ушло время: сервер ходит в Kong по внутреннему имени `http://kong:8000`, а браузер — по `https://api.kv12chat.ru`. У `@supabase/ssr` имя cookie выводится из хоста, поэтому имя auth-cookie зафиксировано (`sb-kv12chat-auth-token`) в `client.ts`/`server.ts`/`proxy.ts` — иначе токен не «переезжает» между серверным и клиентским рендером и получается цикл редиректов.

### Локальная разработка против прода

| | Локально | Прод |
| --- | --- | --- |
| Supabase | `supabase start` (Docker) | Docker Compose в `infra/` |
| API URL | `http://localhost:54321` | `https://api.kv12chat.ru` (через Kong) |
| Next.js | `pnpm dev` | сборка на самом VPS |
| TLS | нет | Caddy + Let's Encrypt |

## 4. Как сделаны ключевые фичи

**Заявка на регистрацию → одобрение → вход.** Самостоятельной регистрации нет (так привычнее для закрытого сообщества и проще модерация). Поток:
1. Гость заполняет форму на `/register` (ФИО, телефон, email, адрес/№ участка, два согласия по 152-ФЗ). `POST /api/registration-request` пишет строку в `registration_requests` (RLS — только админ читает), с дедупликацией по телефону за 24ч и фиксацией согласия.
2. Админ видит заявку в `/admin/registration-requests`, жмёт «Создать пользователя» — открывается форма с предзаполненными данными. Выбирает участок, получает **сгенерированный временный пароль**. `POST /api/admin/people` зовёт `auth.admin.createUser()` (service role) и вставляет строку в `profiles`; заявка помечается закрытой.
3. Житель входит на `/login` по email **или** телефону + временный паролю. На первом входе — экран приветствия/установки PWA.

**Сообщения в реальном времени.** Клиент (`MessageList`/`ThreadPane`) при отправке делает оптимистичную вставку и `POST /api/messages`. Параллельно `subscribeToChannel()` (`@/lib/realtime.ts`) слушает `postgres_changes` по каналу — у собеседника сообщение появляется без перезагрузки. Реакции (`reactions`) и голоса (`poll_votes`) приходят теми же подписками.

**Обсуждения (треды).** Корневое сообщение — `thread_root_id IS NULL`. Ответ ставит `thread_root_id` на корень. На мобильном тред открывается на весь экран, на десктопе — панелью справа.

**Опросы.** Кнопка в меню композера открывает создание голосования; `polls` + `poll_options` + `poll_votes` (PK из трёх полей не даёт проголосовать дважды). Результаты обновляются у всех в реальном времени.

**Вложения-картинки.** Клиент сжимает изображение в canvas до ≤1 МБ, грузит через `POST /api/uploads` в бакет `files`, затем прикрепляет к сообщению. Для приватных файлов отдаётся signed URL на 1 час.

**Редактирование.** Меню сообщения → «Редактировать» → композер с прежним текстом → `PATCH /api/messages/:id`, проставляется `edited_at`, в UI — пометка «(изменено)». Удаление — мягкое (`deleted_at`), история сохраняется.

**Карта посёлка.** `plots` хранит полигоны участков (JSONB-координаты) поверх растровой карты-подложки (SVG-оверлей). Поиск по ФИО/№ участка/адресу: при одном совпадении карта сама фокусируется на участке и показывает его жильцов. Зум/пан/пинч работают на телефоне.

**Push + PWA.** `manifest.ts` + service worker (`public/sw.js`) дают установку на домашний экран и бейдж с числом непрочитанных на иконке. Подписка Web Push (VAPID) хранится в `push_subscriptions`. Триггеры на вставку сообщения кладут задачи в `notification_queue`; `pg_cron` каждые 10с зовёт Edge Function, которая рассылает push и помечает отправленное.

## 5. Разработка и развёртывание

**Разработка.** Вся работа над проектом велась в паре с ИИ-ассистентом (Claude Code): человек формулирует задачу обычным языком и проверяет результат, ассистент пишет схему, RLS, роуты, компоненты. UI/UX-требования удобно описывать прозой — это как раз та часть, которую легко поставить даже не-технарю (см. примеры промптов в §6).

**Деплой — намеренно примитивный, без CI:**

```
ssh deploy@VPS 'cd kv12chat && bash infra/scripts/deploy.sh'
```

`deploy.sh` на самом VPS: `git pull` → накат новых SQL-миграций → `docker compose build nextjs` (Next.js собирается **на сервере**, образ никуда не публикуется) → `docker compose up -d nextjs`. Простой ~5 секунд. Ни GitHub Actions, ни GHCR, ни Vercel.

Этот же код без изменений хостится и в облаке (Supabase Cloud + Vercel) — то есть vendor-lock отсутствует: можно начать в облаке, а потом «съехать» на свой сервер (что в этом проекте и сделали, т.к. в РФ `*.supabase.co`/`*.vercel.app` блокируются).

## 6. Реальные промпты автора (по темам)

Извлечены из истории сессий Claude Code этого проекта (≈254 человеческих промпта, выбрано 12 показательных). Цитаты дословные.

### PWA вместо нативного приложения
> Я могу сделать браузерное приложение, которое юзеры смогут сохранить на телефон как иконку на рабочем столе, и получать с него пуш-уведомлния?

> мне не очень нравится иконка приложения. Особенно в верхней панели телефона, где показываются push-уведомления: там просто белый квадрат. Давай попробуем заменить иконку на число 12 стилизованное под электронный циферблат с толстыми линиями

### Backend на Supabase/BaaS
> для чего нам в этом проекте supabase?

> а как работает наша кастомная авторизация по двум полям: телефон или email? Supabase умеет что-то такое из коробки?

### RLS / безопасность данных
> Немного измени права в каналах. В публичных каналах все могут всё: читать и писать. В приватных тоже: участники этого канала могут всё. И только один служебный канал "объявления" имеет особенность: в нём писать новые сообщения могут только админы, но все остальные жители могут их комментировать в тредах. Кстати, замени в UI везде слово "тред" на "обсуждение".

> этот чат хранит персональные данные пользователей: имена, телефоны, адреса. Значит я попадаю под действие какого-то там федерального закона. Что я должен с этим делать? Достаточно ли просто при подаче заявки на регистрацию нового пользователя просить его нажать галочку "даю согласие на обработку и хранение персональных данных"?

### Realtime / WebSockets
> похоже, в этом приложении не работают вебсокеты

> websocket не работает. Все запросы на wss://api.kv12chat.ru/realtime/v1/websocket получают ответ 403

### Web Push + pg_cron
> мы можем на иконке чата на рабочем столе телефона отобразить количество новых сообщений?

### Self-hosting и деплой без CI
> Использование supabase в качестве бэкенда не подошло: трафик из России к нему блокируется. Какая может быть альтернатива?

> а можно вообще обойтись без github actions?

### UI/UX обычным языком
> Давай сделаем карту на странице "карта" масштабируемой, чтобы на телефоне её можно было приближать, удалять, скроллить в разные стороны. В итоге лэйаут такой: сверху строка поиска — под ней найденные результаты — под ними карта — а под ней информация о конкретном выбранном на карте участке. Если поиск вернул ровно один участок — карта сразу фокусируется на нём […]

> проблемы мобильного UI отправки сообщений: кнопка отправки располагается чуть ниже поля ввода, надо выровнять центры; поле ввода не растёт в высоту когда я много печатаю; две иконки "файл" и "опрос" вместе занимают слишком много места — объедини их в одно контекстное меню […]

## 7. Что забрать с хакатона

1. **Бэкенд для типовых задач можно не писать.** Auth, права (RLS), CRUD, realtime, файлы, cron, очередь уведомлений, полнотекстовый поиск — всё это даёт Supabase поверх одного Postgres. Остаётся UI и немного бизнес-логики в тонких роутах.
2. **PWA закрывает большинство потребностей мобильного приложения**: push, установка на домашний экран, бейдж непрочитанных, офлайн-оболочка, WebSocket — без App Store/Google Play и отдельной кодовой базы.
3. **Нет vendor-lock.** Один и тот же код запускается и в облаке (Supabase+Vercel), и на своём VPS. CI/CD — либо полноценный через GitHub Actions, либо «scp + deploy.sh».
4. **Главная оставшаяся работа — UI/UX**, а её удобно ставить обычным языком. Это смещает центр тяжести проекта с инфраструктуры на продукт.
